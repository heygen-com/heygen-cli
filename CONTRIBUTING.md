# Contributing

## Prerequisites

- Go 1.23+
- [golangci-lint](https://golangci-lint.run/) (for `make lint`)

## Build and Test

```bash
make build    # build to bin/heygen
make install  # install to $GOPATH/bin/heygen
make test     # all tests (mocked, no API key needed)
make lint     # golangci-lint
make clean    # remove build artifacts
```

Smoke test against the real API:

```bash
HEYGEN_API_KEY=<key> ./bin/heygen video list --limit 2
```

## Project Layout

```
cmd/heygen/        Cobra commands, hand-written enhancements, entry point
internal/analytics/ PostHog usage tracking (opt-out)
internal/auth/     Credential resolution (env var → file)
internal/client/   HTTP client, executor, retry logic
internal/command/  Spec, Invocation, Groups, Column types
internal/config/   Config provider (env → file → defaults)
internal/errors/   CLIError, APIError, exit codes
internal/output/   Formatters (JSON, Human/TUI)
internal/paths/    Config and credential file paths
gen/               GENERATED — do not hand-edit
codegen/           OpenAPI → Go codegen pipeline
codegen/examples/  Per-group example YAML files
scripts/           Install script, CI helpers
```

### Key separation

- **`gen/`** — auto-generated `command.Spec` structs. Deleted and regenerated by `make generate`. Never put logic here.
- **`cmd/heygen/`** — hand-written code: custom commands (`auth login`, `config`, `video download`), poll configs, column definitions, flag constraints. This is where behavioral enhancements go.
- **`internal/`** — shared infrastructure. Commands never call the HTTP client directly — they produce a `Spec + Invocation` and the framework's executor handles the rest.

## Regenerating Commands from the OpenAPI Spec

When a new v3 endpoint ships in the API:

```bash
make generate SPEC=/path/to/external-api.json
```

This reads the spec, generates command definitions in `gen/`, and formats the output. Review the diff, then commit the generated files.

Every command must have usage examples. If a new endpoint lacks examples, codegen fails. Add examples to `codegen/examples/{group}.yaml`.

## Adding Hand-Written Enhancements

Generated commands are pure data. To add behavior on top:

**Poll config** (`--wait` support) — register in `cmd/heygen/poll_configs.go`:

```go
"video/create": {
    StatusEndpoint: "/v3/videos/{video_id}",
    StatusField:    "data.status",
    TerminalOK:     []string{"completed"},
    TerminalFail:   []string{"failed"},
    IDField:        "data.video_id",
    HintCommand:    "video get",
},
```

**TUI columns** (`--human` table layout) — register in `cmd/heygen/columns.go`:

```go
"video/list": {
    {Header: "ID", Field: "id"},
    {Header: "Title", Field: "title"},
    {Header: "Status", Field: "status"},
    {Header: "Created", Field: "created_at"},
},
```

**Custom commands** (not in the API spec) — add a new file in `cmd/heygen/` and register in `root.go`. See `video_download.go` for an example.

## Errors

All command errors must use the `CLIError` type:

- `clierrors.New()` — exit code 1 (general)
- `clierrors.NewAuth()` — exit code 3 (auth)
- `clierrors.NewUsage()` — exit code 2 (bad flags/args)
- `clierrors.FromAPIError()` — maps HTTP status to exit code

All error output goes through `formatter.Error()`.

## Testing

All tests use `httptest.Server` mocks — no real API calls, no API key needed.

Command tests use `runCommand()` from `cmd/heygen/testutil_test.go`, which creates a fresh Cobra tree, captures stdout/stderr/exit code, and renders errors through the formatter.
